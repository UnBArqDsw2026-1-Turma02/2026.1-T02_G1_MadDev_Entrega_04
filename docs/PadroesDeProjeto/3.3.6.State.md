# 3.3.2. State (Comportamental)
 
## 1. Introdução
 
O **State** é um padrão comportamental do GoF que **permite que um objeto altere seu comportamento quando seu estado interno muda**, fazendo com que pareça ter mudado de classe. No projeto MadDev, o State resolve o crescimento descontrolado do `_physics_process` do player: em vez de um bloco de `if/else` que cresce a cada nova mecânica (idle, movimento, dash, ataque), a lógica de cada estado é encapsulada em um `Resource` independente. Adicionar ou modificar um estado não exige alterar `_physics_process`.
 
### Quando usar
 
- Quando o comportamento de um objeto depende fortemente de seu estado interno e muda em tempo de execução.
- Quando há muitos condicionais `if/else` ou `match` que verificam flags de estado espalhadas pelo código.
- Quando adicionar um novo estado não deve exigir modificar os estados existentes.

### PlayerState — interface base
 
```gdscript
extends Resource
class_name PlayerState
 
const STATE_IDLE   := &"idle"
const STATE_MOVE   := &"move"
const STATE_DASH   := &"dash"
const STATE_ATTACK := &"attack"
const STATE_NONE   := &""
 
func enter(player: CharacterBody2D) -> void: pass
func update(delta: float, player: CharacterBody2D) -> void: pass
func exit(player: CharacterBody2D) -> void: pass
func get_transition(player: CharacterBody2D) -> StringName: return STATE_NONE
```
 
### Tabela de estados e transições
 
| Estado | Entra quando | Transições possíveis |
|---|---|---|
| `IdleState` | Sem input de movimento | → Move (input detectado), → Dash (dash + can_dash) |
| `MoveState` | Input direcional ativo | → Idle (sem input), → Dash (dash + can_dash) |
| `DashState` | Dash pressionado + cooldown livre | → Move (timer expira + input), → Idle (timer expira + sem input) |
| `AttackState` | Botão de ataque pressionado | → Idle (timer interno expira) |
 
### FSM no player.gd
 
```gdscript
var _states: Dictionary = {}
var _current_state: PlayerState = null
 
func _setup_state_machine() -> void:
    _states[PlayerState.STATE_IDLE]   = IdleState.new()
    _states[PlayerState.STATE_MOVE]   = MoveState.new()
    _states[PlayerState.STATE_DASH]   = DashState.new()
    _states[PlayerState.STATE_ATTACK] = AttackState.new()
    _change_state(PlayerState.STATE_IDLE)
 
func _physics_process(delta: float) -> void:
    _current_state.update(delta, self)
    var next: StringName = _current_state.get_transition(self)
    if next != PlayerState.STATE_NONE:
        _change_state(next)
    move_and_slide()
```
 
### API pública do player para os estados
 
| Método | Descrição |
|---|---|
| `get_move_input() -> Vector2` | Lê o vetor de movimento via InputAdapter |
| `is_dash_just_pressed() -> bool` | Verifica se dash foi pressionado neste frame |
| `can_dash() -> bool` | Retorna se o cooldown de dash está disponível |
| `start_dash_cooldown()` | Inicia o cooldown sem bloquear a FSM com `await` |

## 3. Funcionamento
 
```gdscript
# Antes: _physics_process com flags e condicionais
func _physics_process(delta):
    if _is_dashing:
        move_and_slide()
        return
    _move_direction = Input.get_vector(...)
    velocity = _move_direction * move_speed
    if Input.is_action_just_pressed("dash") and _can_dash:
        _execute_dash()   # usava await internamente
    move_and_slide()
 
# Depois: _physics_process de 4 linhas
func _physics_process(delta: float) -> void:
    _current_state.update(delta, self)
    var next := _current_state.get_transition(self)
    if next != PlayerState.STATE_NONE:
        _change_state(next)
    move_and_slide()
```
 
O `DashState` usa um timer interno (`_elapsed += delta`) para controlar a duração do dash, eliminando a necessidade de `await` que incompatibilizava com a FSM.
 
### Benefícios
 
- **Isolamento:** modificar o dash não afeta a lógica de movimento ou ataque.
- **Extensibilidade:** novo estado = nova classe Resource, sem tocar em `_physics_process`.
- **Sem await:** temporização feita por delta acumulado nos próprios estados.
- **Testabilidade:** cada estado pode ser testado isoladamente passando um player mockado.

## 4. Metodologia
 
A construção desta documentação seguiu três etapas, tomando como fonte primária o código da branch `feat/8-state-player-fsm`:
 
1. **Levantamento da FSM.** Leitura de `player_state.gd` (interface base) e dos 4 estados concretos para mapear as 4 funções de ciclo de vida (`enter`, `update`, `exit`, `get_transition`) e o uso de constantes `StringName` para identificar estados, evitando strings literais espalhadas pelo código.
2. **Análise da integração com o Adapter.** Verificação de que os estados não chamam `Input.*` diretamente — usam a API pública do player (`get_move_input()`, `is_dash_just_pressed()`) que internamente delega ao `InputAdapter` (padrão #7), mantendo os dois padrões ortogonais.
3. **Trace do DashState.** Inspeção do timer interno `_elapsed` e do mecanismo `start_dash_cooldown()` (que usa `create_timer().timeout.connect()`) para confirmar a ausência de `await` — bloqueio incompatível com FSMs baseadas em `_physics_process`.
**Ferramentas utilizadas:** Godot 4 (validação de `CharacterBody2D.move_and_slide()` com FSM), Git (branch `feat/8-state-player-fsm`), Markdown + Docsify (apresentação).
