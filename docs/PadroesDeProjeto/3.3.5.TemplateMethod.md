# 3.3.1. Template Method (Comportamental)

## 1. Introdução

O **Template Method** é um padrão comportamental do GoF que **define o esqueleto de um algoritmo na classe base e permite que as subclasses sobrescrevam etapas específicas** sem alterar a sequência geral. No projeto MadDev, a sequência de ataque dos inimigos (`can_attack` → `prepare_attack` → `execute_attack` → `cooldown`) é definida em `EnemyBase.attack_sequence()`, e cada inimigo concreto (`EnemyMelee`, `EnemyRanged`) sobrescreve apenas as etapas que diferem.

### Quando usar

- Quando vários objetos seguem uma mesma sequência de operações com pequenas variações.
- Quando se quer evitar duplicação de "fluxo" entre classes irmãs.
- Quando o conjunto de etapas (e sua ordem) deve ser controlado pela classe base.

### O método template em `enemy_base.gd`

```gdscript
@export var attack_cooldown_time: float = 1.0
var _attack_cooldown_remaining: float = 0.0

func attack_sequence() -> void:
    if _attack_cooldown_remaining > 0:
        return
    if not can_attack():
        return
    prepare_attack()
    execute_attack()
    cooldown()

# Hooks com implementações padrão
func can_attack() -> bool:
    return true

func prepare_attack() -> void:
    pass

func execute_attack() -> void:
    pass

func cooldown() -> void:
    _attack_cooldown_remaining = attack_cooldown_time
```

A sequência é **fixa** — o concreto não controla a ordem, apenas o conteúdo de cada etapa.

### Etapas do template

| Etapa | Quem implementa | Descrição |
|---|---|---|
| `can_attack()` | **Hook** (sobrescreve quando precisa filtrar) | Decide se o ataque pode acontecer agora (distância, visão, estado vivo). |
| `prepare_attack()` | **Hook opcional** | Animação/som de preparação. |
| `execute_attack()` | **Primitive** (deve sobrescrever) | Aplica o efeito real (dano corpo-a-corpo, projétil, etc.). |
| `cooldown()` | Padrão raramente sobrescrita | Inicia o tempo de recarga. |

A sigla GoF "Hook" é uma etapa que tem implementação padrão sensata e pode ser sobrescrita; "Primitive" é uma etapa que **precisa** ser sobrescrita para ter comportamento útil.

### Implementações concretas

#### EnemyMelee — corpo-a-corpo

```gdscript
class_name EnemyMelee
extends EnemyBase

@export var melee_range: float = 40.0

func can_attack() -> bool:
    if _is_dead or _player == null:
        return false
    return global_position.distance_to(_player.global_position) <= melee_range

func prepare_attack() -> void:
    print(name, " se prepara para atacar!")

func execute_attack() -> void:
    print(name, " ATACOU COM A FORÇA BRUTA!")
    _apply_damage_to_player(attack_damage * 2)
```

Sobrescreve as 3 primeiras etapas: filtra por distância, anuncia o ataque e dobra o dano base no corpo-a-corpo.

#### EnemyRanged — à distância (via Object Pool)

```gdscript
class_name EnemyRanged
extends EnemyBase

@export var max_shoot_range: float = 300.0

func can_attack() -> bool:
    if _is_dead or _player == null or _pool == null:
        return false
    return global_position.distance_to(_player.global_position) <= max_shoot_range

func execute_attack() -> void:
    var dir: Vector2 = global_position.direction_to(_player.global_position)
    _pool.get_projectile(global_position, dir, self)
```

Sobrescreve `can_attack` (precisa do pool de projéteis disponível) e `execute_attack` (dispara via `ProjectilePool`, integrando com o padrão [Object Pool (3.1.4)](/PadroesDeProjeto/3.1.4.ObjectPool.md)).

## 3. Funcionamento

A sequência de ataque é chamada do loop de comportamento do inimigo:

```gdscript
# Dentro do _physics_process do EnemyBase:
attack_sequence()
_update_cooldown(delta)
```

Em runtime, o fluxo para um `EnemyMelee` é:

```
attack_sequence()
    ├── _attack_cooldown_remaining > 0 ?  → return (ainda em cooldown)
    ├── can_attack() → distância <= 40 ?  → false: return
    ├── prepare_attack() → print "se prepara"
    ├── execute_attack() → dano * 2 no player
    └── cooldown() → reseta _attack_cooldown_remaining
```

Para um `EnemyRanged`:

```
attack_sequence()
    ├── cooldown ativo ?  → return
    ├── can_attack() → pool disponível + distância <= 300 ?  → false: return
    ├── prepare_attack() → (padrão, vazio)
    ├── execute_attack() → ProjectilePool.get_projectile(...)
    └── cooldown() → reseta
```

### Benefícios

- **Sequência inviolável:** subclasses não conseguem (acidentalmente) executar antes de verificar `can_attack` ou pular o `cooldown`.
- **Duplicação eliminada:** o loop `verifica → prepara → executa → recarrega` vive em um único lugar.
- **Variação concentrada:** cada inimigo concreto fica curto, mostrando só o que muda.
- **Extensível:** adicionar um novo tipo de inimigo é criar uma nova classe que estende `EnemyBase` e sobrescrever as etapas relevantes.

## 4. Metodologia

A construção desta documentação seguiu três etapas, tomando como fonte primária o código da branch `game`:

1. **Levantamento do método template.** Leitura de `enemy_base.gd` (linhas 90–97) para mapear `attack_sequence()`, a ordem fixa das 4 etapas e o guard de `_attack_cooldown_remaining`.

2. **Mapeamento das implementações padrão.** Identificação de quais etapas têm corpo padrão útil (`cooldown`, `can_attack` retornando `true`) versus quais são "vazias" e precisam de override (`prepare_attack`, `execute_attack`).

3. **Análise das classes concretas.** Leitura de `enemy_melee.gd` e `enemy_ranged.gd` para confirmar quais etapas cada uma sobrescreve e como cada concreto difere — incluindo a integração do `EnemyRanged` com o Object Pool.
