# Padrão de Projeto: Multiton (Registry de Dificuldade)
---

## 1. Introdução

O **Multiton** é uma variação do padrão Singleton que mantém um **conjunto fixo de instâncias únicas indexadas por chave**, em vez de uma única instância global. Enquanto o Singleton garante "uma instância", o Multiton garante "uma instância por chave".

No projeto MadDev, esse padrão é usado para representar **modos de dificuldade** (`easy`, `normal`, `hard`): cada chave aponta para uma instância imutável de `DifficultyConfig`, e qualquer parte do código (IA de inimigo, sistema de loot, balanceamento) acessa a mesma configuração de forma estática.

### Quando usar?

- Quando existe um número fixo e conhecido de variantes de um mesmo conceito.
- Quando todas as variantes precisam ser acessíveis globalmente, sem acoplamento com nós da SceneTree (`get_node`).
- Quando a identidade da instância importa: duas chamadas com a mesma chave devem retornar **o mesmo objeto na memória**.

> Nota: o Multiton **não consta** no catálogo original do GoF (Gamma et al., 1994). É uma variação consolidada pela comunidade, frequentemente classificada como padrão criacional derivado do Singleton.

### Estrutura

```
DifficultyRegistry  ──  _instances: Dictionary  ──┬─>  "easy"   : DifficultyConfig
                                                  ├─>  "normal" : DifficultyConfig
                                                  └─>  "hard"   : DifficultyConfig
```

## 2. Multiton no GDScript: static var

No Godot 4, há duas estratégias para garantir instâncias globais:

1. **Autoload** — registrar a cena/script em `project.godot` como singleton nativo (usado pelo `StudentProfileRegistry`, ver [Singleton (3.1.5)](/PadroesDeProjeto/3.1.5.Singleton.md)).
2. **`static var` em classe `RefCounted`** — manter o estado em variáveis estáticas de uma classe que nunca é instanciada (usado pelo `DifficultyRegistry`).

O `DifficultyRegistry` adota a segunda estratégia: a classe é `RefCounted` mas todos os membros são `static`, então **nenhuma instância da classe é criada** — o dicionário de instâncias vive como propriedade da própria classe.

### Vantagens e limitações

| Aspecto | `static var` (este caso) | Autoload |
|---|---|---|
| Configuração em `project.godot` | Não requer | Requer registro explícito |
| Acesso | `DifficultyRegistry.get_instance(key)` | `StudentProfileRegistry.get_profile(key)` |
| Instância da classe | Nenhuma | Uma (na raiz da SceneTree) |
| Persistência entre cenas | Garantida (vive no espaço da classe) | Garantida (filho da SceneTree) |
| Thread safety | Responsabilidade do dev | Responsabilidade do dev |

## 3. Implementação no Projeto

### 3.1 DifficultyConfig (Resource)

**Arquivo:** `jogo/scripts/resources/difficulty_config.gd`
**Responsabilidade:** estrutura de dados (`Resource`) com os modificadores de balanceamento de uma dificuldade.

```gdscript
class_name DifficultyConfig
extends Resource

@export var enemy_hp_modifier: float = 1.0
@export var enemy_speed_modifier: float = 1.0
@export var item_drop_rate: float = 1.0

func _init() -> void:
    pass

func setup(hp: float, speed: float, drop: float) -> DifficultyConfig:
    enemy_hp_modifier = hp
    enemy_speed_modifier = speed
    item_drop_rate = drop
    return self
```

| Campo | Tipo | Significado |
|---|---|---|
| `enemy_hp_modifier` | `float` | Multiplicador aplicado à vida dos inimigos |
| `enemy_speed_modifier` | `float` | Multiplicador aplicado à velocidade dos inimigos |
| `item_drop_rate` | `float` | Multiplicador aplicado à taxa de drop de itens |

O método `setup()` retorna `self` para permitir encadeamento fluente no momento do registro.

### 3.2 DifficultyRegistry (a classe Multiton)

**Arquivo:** `jogo/scripts/resources/difficulty_registry.gd`
**Responsabilidade:** ponto de acesso global ao dicionário de instâncias `DifficultyConfig` indexado por `StringName`.

```gdscript
class_name DifficultyRegistry
extends RefCounted

# O dicionário estático que guarda as instâncias únicas nomeadas
static var _instances: Dictionary = {}

## Retorna a instância associada à chave. Se não existir, cria uma com valores padrão.
static func get_instance(key: StringName) -> DifficultyConfig:
    if not _instances.has(key):
        _instances[key] = DifficultyConfig.new()
    return _instances[key]

## Permite registrar ou sobrescrever explicitamente uma configuração de dificuldade
static func register_difficulty(key: StringName, config: DifficultyConfig) -> void:
    _instances[key] = config
```

| Método | Descrição |
|---|---|
| `get_instance(key)` | Retorna a instância associada à chave. Se a chave não existir, cria um `DifficultyConfig` com valores padrão (1.0, 1.0, 1.0) e armazena. |
| `register_difficulty(key, config)` | Registra ou sobrescreve a configuração da chave. |

### 3.3 Registro no GameManager._ready()

**Arquivo:** `jogo/scripts/autoloads/game_manager.gd`

As três dificuldades padrão são criadas e registradas durante a inicialização do `GameManager` (que é Autoload e portanto roda uma única vez no boot):

```gdscript
func _ready() -> void:
    var easy: DifficultyConfig = DifficultyConfig.new()
    easy.setup(0.75, 0.85, 1.5)

    var normal: DifficultyConfig = DifficultyConfig.new()
    normal.setup(1.0, 1.0, 1.0)

    var hard: DifficultyConfig = DifficultyConfig.new()
    hard.setup(1.5, 1.2, 0.7)

    DifficultyRegistry.register_difficulty(&"easy", easy)
    DifficultyRegistry.register_difficulty(&"normal", normal)
    DifficultyRegistry.register_difficulty(&"hard", hard)
```

| Chave | `enemy_hp_modifier` | `enemy_speed_modifier` | `item_drop_rate` |
|---|---|---|---|
| `&"easy"` | 0.75 | 0.85 | 1.5 |
| `&"normal"` | 1.0 | 1.0 | 1.0 |
| `&"hard"` | 1.5 | 1.2 | 0.7 |

### 3.4 Exemplo de uso (consumidor)

Qualquer script do jogo pode acessar a configuração da dificuldade vigente de forma estática e tipada, sem precisar caçar nós na árvore da cena:

```gdscript
extends CharacterBody2D

var max_health: float = 100.0

func _ready() -> void:
    # 1. Puxa a configuração do modo Hard de forma segura e tipada
    var config: DifficultyConfig = DifficultyRegistry.get_instance(&"hard")

    # 2. Aplica o modificador de vida da dificuldade
    max_health *= config.enemy_hp_modifier
```

## 4. Comparação: DifficultyRegistry vs. StudentProfileRegistry

O projeto MadDev contém **duas implementações distintas do padrão Multiton**, com propósitos e mecânicas diferentes:

| Aspecto | `DifficultyRegistry` (este documento) | `StudentProfileRegistry` (ver [Singleton (3.1.5)](/PadroesDeProjeto/3.1.5.Singleton.md)) |
|---|---|---|
| Mecanismo de Singleton | `static var` em classe `RefCounted` | Autoload do Godot |
| Origem das instâncias | Criadas em código (`DifficultyConfig.new()` + `setup`) | Carregadas do disco (`.tres` em `res://resources/profiles/`) |
| Chaves | Definidas no `GameManager._ready()` (`easy`, `normal`, `hard`) | Definidas em dicionário privado `_PROFILE_PATHS` (`Calouro`, `Veterano`, `Jubilado`, `Cara da Atletica`) |
| Sobrescrita em runtime | Sim (`register_difficulty`) | Não (chaves fixas no código) |
| Tipo dos valores | Dados de balanceamento (`Resource` simples) | Perfis completos de personagem |

A coexistência das duas abordagens mostra que **Multiton é mais uma intenção (N instâncias indexadas por chave) do que uma implementação rígida**: a escolha entre `static var` e Autoload depende de o ciclo de vida da entidade (configuração estática vs. dados persistidos em disco) e da necessidade ou não de sobrescrita em runtime.

## 5. Validação

O arquivo `jogo/scripts/teste_multiton.gd` valida a propriedade central do Multiton — **idempotência por chave**:

```gdscript
extends Node

func _ready() -> void:
    await get_tree().process_frame

    print("\n--- INICIANDO TESTE DO MULTITON ---")

    var easy_mode: DifficultyConfig = DifficultyRegistry.get_instance(&"easy")
    print("Modo Fácil -> Vida: %s, Velocidade: %s" % [easy_mode.enemy_hp_modifier, easy_mode.enemy_speed_modifier])

    var hard_mode: DifficultyConfig = DifficultyRegistry.get_instance(&"hard")
    print("Modo Difícil -> Vida: %s, Drop Rate: %s" % [hard_mode.item_drop_rate, hard_mode.enemy_hp_modifier])

    var hard_mode_copy = DifficultyRegistry.get_instance(&"hard")

    if hard_mode == hard_mode_copy:
        print("✅ SUCESSO: O Multiton retornou a mesma instância de memória!")
    else:
        print("❌ ERRO: O Multiton criou instâncias duplicadas para a mesma chave.")
```

O `await get_tree().process_frame` garante que o `_ready()` do `GameManager` (que faz o registro das três dificuldades) já tenha sido executado antes do teste consultar o registry.

A comparação `==` entre dois `Resource` em Godot verifica **identidade de referência** — ou seja, esse teste confirma que `get_instance(&"hard")` devolve **o mesmo objeto na memória** em ambas as chamadas, e não dois objetos com valores iguais.

## 6. Metodologia

A construção desta documentação seguiu três etapas, todas tomando como fonte primária o código da branch `origin/issue-11-registry-perfis`:

1. **[Levantamento dos arquivos da implementação](#_3-implementação-no-projeto).** Leitura de `jogo/scripts/resources/difficulty_config.gd` e `jogo/scripts/resources/difficulty_registry.gd` para mapear a estrutura (Resource e classe estática), os métodos públicos e a responsabilidade de cada arquivo.

2. **[Trace do registro e do uso](#_33-registro-no-gamemanager_ready).** Leitura de `jogo/scripts/autoloads/game_manager.gd` para identificar o ponto de registro das três dificuldades padrão no `_ready()`, e de `jogo/scripts/teste_multiton.gd` para entender o critério de [validação](#_5-validação) (identidade de referência entre chamadas com a mesma chave).

3. **[Diferenciação Multiton via static var vs. via Autoload](#_4-comparação-difficultyregistry-vs-studentprofileregistry).** Comparação com a implementação alternativa do mesmo padrão (`StudentProfileRegistry`, documentada em [Singleton (3.1.5)](/PadroesDeProjeto/3.1.5.Singleton.md)) para evidenciar os trade-offs entre as duas abordagens dentro do próprio projeto.

**Ferramentas utilizadas:** Godot 4 (leitura do código e do `project.godot`), Git (versionamento e leitura de branches paralelas), Markdown + Docsify (apresentação).
