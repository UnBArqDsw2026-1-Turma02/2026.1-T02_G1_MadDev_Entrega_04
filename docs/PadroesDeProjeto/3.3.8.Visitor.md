# 3.3.4. Visitor (Comportamental)
 
## 1. Introdução
 
O **Visitor** é um padrão comportamental do GoF que **permite adicionar operações a objetos sem modificar suas classes**, separando o algoritmo da estrutura de dados sobre a qual ele opera. No projeto MadDev, o Visitor resolve a necessidade de coletar estatísticas de diferentes tipos de nó em uma sala (inimigos, itens, projéteis, tilemaps) para fins de debug e balanceamento: em vez de adicionar lógica de relatório em cada classe base, o `StatsVisitor` percorre a sala e coleta os dados. As classes visitadas expõem apenas um `accept(visitor)` — sem conhecer o que o visitor vai fazer com as informações.
 
### Quando usar
 
- Quando uma operação deve ser realizada sobre objetos de uma estrutura heterogênea sem alterar suas classes.
- Quando a mesma estrutura de objetos deve suportar múltiplas operações independentes (relatório, validação, exportação).
- Quando adicionar novos tipos de visitante não deve exigir modificar os nós visitados.

> **Nota:** `TileMapLayer` é uma classe built-in do Godot — não é possível adicionar `accept()` sem subclasse. O `RoomStatsReport` a visita diretamente via `visitor.visit_tilemap(node)`.
 
### StatsVisitor — coleta de dados
 
```gdscript
extends RefCounted
class_name StatsVisitor
 
var _report: Dictionary = {}
 
func _init() -> void:
    _report = {
        "enemy_count": 0,         "total_enemy_hp": 0,
        "total_enemy_max_hp": 0,  "item_count": 0,
        "projectile_count": 0,    "tilemap_layer_count": 0,
        "total_tilemap_cells": 0,
    }
 
func visit_enemy(enemy: EnemyBase) -> void:
    _report["enemy_count"]        += 1
    _report["total_enemy_hp"]     += enemy.current_health
    _report["total_enemy_max_hp"] += enemy.max_health
 
func visit_item(_item: Node) -> void:
    _report["item_count"] += 1
 
func visit_projectile(_projectile: Node) -> void:
    _report["projectile_count"] += 1
 
func visit_tilemap(tilemap: TileMapLayer) -> void:
    _report["tilemap_layer_count"] += 1
    _report["total_tilemap_cells"] += tilemap.get_used_cells().size()
 
func get_report() -> Dictionary:
    return _report.duplicate()
```
 
### Dados coletados pelo relatório
 
| Campo | Tipo | Descrição |
|---|---|---|
| `enemy_count` | `int` | Total de inimigos vivos na sala |
| `total_enemy_hp` | `int` | Soma dos HP atuais de todos os inimigos |
| `total_enemy_max_hp` | `int` | Soma dos HP máximos |
| `item_count` | `int` | Total de itens presentes |
| `projectile_count` | `int` | Total de projéteis ativos |
| `tilemap_layer_count` | `int` | Camadas TileMapLayer encontradas |
| `total_tilemap_cells` | `int` | Células usadas em todas as camadas |
 
### accept() nas classes base
 
```gdscript
## Em EnemyBase — Double Dispatch via Object (evita dependência circular)
func accept(visitor: Object) -> void:
    visitor.visit_enemy(self)
 
## Em ItemBase (snippet)
func accept(visitor: Object) -> void:
    visitor.visit_item(self)
 
## Em ProjectileBase (snippet)
func accept(visitor: Object) -> void:
    visitor.visit_projectile(self)
```
 
> **Por que `Object` e não `StatsVisitor`?** EnemyBase usa `StatsVisitor` em `accept()` e StatsVisitor usa `EnemyBase` em `visit_enemy()` — dependência circular que o parser do GDScript não resolve. O tipo `Object` elimina a dependência em tempo de parse, mantendo o comportamento correto em runtime via duck typing.
 
### RoomStatsReport — orquestração por grupos
 
```gdscript
extends Node
class_name RoomStatsReport
 
@export var debug_key: Key = KEY_F1
 
func _input(event: InputEvent) -> void:
    if event is InputEventKey and event.pressed and not event.echo:
        if event.keycode == debug_key:
            _print_report(collect())
 
func collect() -> Dictionary:
    var visitor := StatsVisitor.new()
    for node in get_tree().get_nodes_in_group("enemies"):
        if node.has_method("accept"):
            node.accept(visitor)
    for node in get_tree().get_nodes_in_group("items"):
        if node.has_method("accept"):
            node.accept(visitor)
    for node in get_tree().get_nodes_in_group("projectiles"):
        if node.has_method("accept"):
            node.accept(visitor)
    for node in get_tree().get_nodes_in_group("tilemaps"):
        if node is TileMapLayer:
            visitor.visit_tilemap(node as TileMapLayer)
    return visitor.get_report()
```

## 3. Funcionamento
 
```gdscript
# Adicionar RoomStatsReport como filho da cena Room no editor.
# Pressionar F1 durante o jogo exibe no console:
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
#   ROOM STATS REPORT — 14:32:01
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
#   Inimigos   : 3
#   HP total   : 65 / 90
#   Itens      : 2
#   Projéteis  : 5
#   TileMapLayers : 2  (312 células)
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
 
# Uso via código (ex: em testes de balanceamento):
var stats := $RoomStatsReport.collect()
print(stats["enemy_count"])   # → 3
```
 
Para adicionar um novo tipo visitável, basta: criar a classe com `accept(visitor: Object)` e adicionar os nós ao grupo correspondente no inspetor.
 
### Benefícios
 
- **Não-invasivo:** as classes visitadas expõem apenas `accept()` — sem lógica de relatório acoplada.
- **Extensível:** novo tipo de visitor (ex: `ValidationVisitor`) não exige modificar os nós.
- **Debug por grupos:** `get_nodes_in_group()` dispensa travessia manual da SceneTree.
- **Tecla configurável:** `@export var debug_key` permite mudar o atalho sem editar código.

## 4. Metodologia
 
A construção desta documentação seguiu três etapas, tomando como fonte primária o código da branch `feat/10-visitor-room-stats`:
 
1. **Levantamento do Visitor e Double Dispatch.** Leitura de `stats_visitor.gd` para mapear os 4 métodos `visit_*`, o dicionário interno `_report` e o uso de `RefCounted` (sem SceneTree) como base, e análise do `accept(visitor: Object)` em `EnemyBase` para identificar o Double Dispatch — o tipo `Object` foi necessário para evitar dependência circular com `EnemyBase`.
2. **Análise da orquestração por grupos.** Inspeção de `room_stats_report.gd` para confirmar o uso de `get_tree().get_nodes_in_group()` para localizar nós sem travessia manual, e o tratamento especial de `TileMapLayer` (built-in sem `accept()`) via chamada direta a `visitor.visit_tilemap()`.
3. **Trace da exibição de debug.** Verificação do `_input()` com `event.pressed and not event.echo` para evitar disparo repetido em teclas mantidas pressionadas, e do formato de saída com separadores visuais para legibilidade no console do Godot.
**Ferramentas utilizadas:** Godot 4 (validação de `get_nodes_in_group()` e `TileMapLayer.get_used_cells()`), Git (branch `feat/10-visitor-room-stats`), Markdown + Docsify (apresentação).
