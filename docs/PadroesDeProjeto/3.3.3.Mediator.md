# GameMediator

Ponto central de comunicação entre sistemas do jogo. Em vez de nós se chamarem diretamente, publicam eventos no mediator que os roteia para quem estiver registrado.

## Quando usar

Use o `GameMediator` quando o receptor é **instanciado em runtime** e precisa gerenciar seu próprio ciclo de vida no sistema de eventos.

Use o `SignalBus` via inspetor quando o receptor é um **nó fixo de cena** com ciclo de vida estável (HUD, PauseMenu, AudioManager).

| Situação | Use |
|---|---|
| Nó criado/destruído em runtime (ex: sala, inimigo) | `GameMediator` |
| Sistema que filtra eventos por dados (ex: só boss) | `GameMediator` |
| Sistema de achievements ou analytics | `GameMediator` |
| Nó fixo de cena (ex: HUD, PauseMenu) | `SignalBus` via inspetor |

## Como usar

### Publicar um evento

```gdscript
GameMediator.notify(self, GameMediator.EVENT_ENEMY_DIED, {"enemy": self})
```

O terceiro argumento é opcional. Use-o para passar dados relevantes ao evento.

### Registrar um handler

```gdscript
func _ready() -> void:
    GameMediator.register(GameMediator.EVENT_ENEMY_DIED, _on_enemy_died)

func _on_enemy_died(sender: Object, event: StringName, data: Dictionary) -> void:
    var enemy = data.get("enemy", sender)
    # ...
```

### Remover o handler

Sempre remova o handler quando o nó sair da árvore para evitar callbacks inválidos.

```gdscript
func _exit_tree() -> void:
    GameMediator.unregister(GameMediator.EVENT_ENEMY_DIED, _on_enemy_died)
```

## Eventos disponíveis

| Constante | Dados (`data`) |
|---|---|
| `EVENT_PLAYER_HEALTH_CHANGED` | `new_health: int`, `max_health: int` |
| `EVENT_PLAYER_DIED` | — |
| `EVENT_ENEMY_SPAWNED` | `enemy: Node` |
| `EVENT_ENEMY_DIED` | `enemy: Node` |
| `EVENT_ROOM_CLEARED` | — |
| `EVENT_RUN_STARTED` | — |
| `EVENT_RUN_ENDED` | `victory: bool` |
| `EVENT_GAME_PAUSED` | `is_paused: bool` |
| `EVENT_SCORE_CHANGED` | `new_score: int` |

## Adicionando um novo evento

1. Declare a constante em `game_mediator.gd`:

```gdscript
const EVENT_ITEM_PICKED_UP: StringName = &"item_picked_up"
```

2. Se sistemas via inspetor precisarem ouvir, adicione o sinal correspondente em `SignalBus` e registre a ponte (bridge) em `_register_signal_bus_bridge()`:

```gdscript
# Em game_mediator.gd
func _register_signal_bus_bridge() -> void:
    # ... registros existentes
    register(EVENT_ITEM_PICKED_UP, _emit_item_picked_up)

func _emit_item_picked_up(_sender: Object, _event: StringName, data: Dictionary) -> void:
    SignalBus.item_picked_up.emit(data.get("item_data", null))
```

A partir daí o evento pode ser usado de duas formas:

- **Sistemas dinâmicos** (criados em runtime): `GameMediator.notify(self, GameMediator.EVENT_ITEM_PICKED_UP, {"item_data": item})` e `GameMediator.register(EVENT_ITEM_PICKED_UP, callback)`.
- **Nós fixos de cena**: conectam-se a `SignalBus.item_picked_up` via inspetor — a bridge cuida do encaminhamento.

## Implementação interna

O `GameMediator` é um Autoload (`extends Node`) registrado em `project.godot`. O estado é mantido em `_handlers: Dictionary` — um mapa de `event: StringName` → `Array[Callable]`.

| Método | Responsabilidade |
| --- | --- |
| `notify(sender, event, data)` | Para cada callback registrado naquele evento, chama `callback.call(sender, event, data)`. A lista é duplicada antes do loop (`callbacks.duplicate()`) para permitir `unregister` durante o despacho sem invalidar o iterador. |
| `register(event, callback)` | Adiciona o callback ao array do evento (cria o array se for o primeiro). Ignora duplicatas e emite warning para `Callable` inválido. |
| `unregister(event, callback)` | Remove o callback do array e, se ficar vazio, remove a entrada do evento. |
| `_register_signal_bus_bridge()` | Chamado em `_ready()`. Registra handlers internos (`_emit_*`) que reenviam 8 dos 9 eventos para sinais correspondentes em `SignalBus`. |

> Observação: `EVENT_ENEMY_SPAWNED` é declarado como constante mas **não** é encaminhado via bridge — só pode ser ouvido por consumidores que se registram diretamente via `GameMediator.register(...)`.

## Metodologia

A construção desta documentação seguiu três etapas, todas tomando como fonte primária o código da branch `game`:

1. **Levantamento dos métodos públicos.** Leitura de `jogo/scripts/autoloads/game_mediator.gd` para mapear a API (`notify`, `register`, `unregister`) e o conjunto de constantes `EVENT_*` declaradas.

2. **Trace da bridge com `SignalBus`.** Análise de `_register_signal_bus_bridge()` e dos 8 métodos `_emit_*` para identificar quais eventos são encaminhados ao `SignalBus` e qual o formato de `data` esperado em cada um.

3. **Diferenciação Mediator vs. SignalBus.** Comparação das duas abordagens de comunicação no projeto para evidenciar quando cada uma é apropriada — base para a seção "Quando usar".
