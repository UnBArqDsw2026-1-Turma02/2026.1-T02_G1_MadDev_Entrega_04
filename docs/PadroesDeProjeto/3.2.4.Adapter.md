# 3.2.2. Adapter (Estrutural)
 
## 1. Introdução
 
O **Adapter** é um padrão estrutural do GoF que **converte a interface de uma classe em outra interface esperada pelos clientes**, permitindo que classes incompatíveis trabalhem juntas. No projeto MadDev, o Adapter resolve o acoplamento direto entre o `player.gd` e o sistema de Input do Godot: em vez de chamar `Input.get_vector()` e `Input.is_action_just_pressed()` diretamente, o player delega toda leitura de controles ao `InputAdapter`, que traduz `InputEvent` bruto para `GameAction` tipado. Isso desacopla o gameplay da API de Input do engine e permite substituir o adapter (por replay, IA ou testes) sem alterar o player.
 
### Quando usar
 
- Quando um sistema depende de uma API externa (ex: `Input` do Godot) e deseja-se isolar essa dependência.
- Quando a interface existente é incompatível com o que o cliente espera receber.
- Quando se quer facilitar testes automatizados trocando a implementação real por um stub.

## 2. Implementação no Projeto
 
### Arquivos
 
| Arquivo | Papel |
|---|---|
| `jogo/scripts/input/game_action.gd` | Dado tipado que representa uma ação de jogo com nome e intensidade. |

| `jogo/scripts/player/player.gd` | Modificado para nunca chamar `Input` diretamente. |
 
Os arquivos foram introduzidos na branch `feat/7-adapter-input`.
 
### GameAction — dado tipado
 
```gdscript
extends Resource
class_name GameAction
 
@export var action_name: StringName = &""
@export var strength: float = 0.0
 
static func make(p_name: StringName, p_strength: float = 1.0) -> GameAction:
    var a := GameAction.new()
    a.action_name = p_name
    a.strength    = p_strength
    return a
 
func is_valid() -> bool:
    return action_name != &"" and action_name != &"none"
```
 
### InputAdapter — mapeamento das 5 ações
 
```gdscript
extends Node
class_name InputAdapter
 
const ACTION_MOVE_UP    := &"move_up"
const ACTION_MOVE_DOWN  := &"move_down"
const ACTION_MOVE_LEFT  := &"move_left"
const ACTION_MOVE_RIGHT := &"move_right"
const ACTION_DASH       := &"dash"
 
const ALL_ACTIONS: Array = [
    ACTION_MOVE_UP, ACTION_MOVE_DOWN,
    ACTION_MOVE_LEFT, ACTION_MOVE_RIGHT, ACTION_DASH,
]
 
## Event-driven: converte um InputEvent para GameAction.
func translate(event: InputEvent) -> GameAction:
    for action_name: StringName in ALL_ACTIONS:
        if event.is_action(action_name):
            return GameAction.make(action_name, event.get_action_strength(action_name))
    return GameAction.make(&"none", 0.0)
 
## Polling: retorna vetor de movimento (WASD + setas + analógico).
func poll_move_vector() -> Vector2:
    return Input.get_vector(ACTION_MOVE_LEFT, ACTION_MOVE_RIGHT,
                            ACTION_MOVE_UP, ACTION_MOVE_DOWN)
 
## Polling: retorna true se a ação foi pressionada neste frame.
func is_just_pressed(action_name: StringName) -> bool:
    return Input.is_action_just_pressed(action_name)
```
 
### API do InputAdapter
 
| Método | Uso | Descrição |
|---|---|---|
| `translate(event)` | Event-driven | Converte um `InputEvent` para `GameAction` |
| `poll_move_vector()` | `_physics_process` | Retorna vetor 2D de movimento |
| `is_just_pressed(action)` | `_physics_process` | Detecta ação pressionada neste frame |
| `is_pressed(action)` | Contínuo | Detecta ação mantida pressionada |
| `get_strength(action)` | Analógico | Retorna intensidade [0.0, 1.0] |
 
### player.gd — uso do Adapter
 
```gdscript
@onready var _input_adapter: InputAdapter = InputAdapter.new()
 
func _ready() -> void:
    add_child(_input_adapter)   # Registra na árvore de cena
 
func _read_move_input() -> Vector2:
    return _input_adapter.poll_move_vector()    # sem Input.*
 
func _read_dash_input() -> bool:
    return _input_adapter.is_just_pressed(InputAdapter.ACTION_DASH)  # sem Input.*
```

## 3. Funcionamento
 
```gdscript
# Antes: Input chamado diretamente em player.gd
func _read_move_input() -> Vector2:
    return Input.get_vector("move_left", "move_right", "move_up", "move_down")
 
func _read_dash_input() -> bool:
    return Input.is_action_just_pressed("dash")
 
# Depois: toda leitura passa pelo adapter
func _read_move_input() -> Vector2:
    return _input_adapter.poll_move_vector()
 
func _read_dash_input() -> bool:
    return _input_adapter.is_just_pressed(InputAdapter.ACTION_DASH)
```
 
Para suportar gamepad analógico ou trocar o layout de controles, basta atualizar as constantes em `InputAdapter` — nenhum código de gameplay muda.
 
### Benefícios
 
- **Isolamento total:** zero chamadas a `Input.*` fora do `InputAdapter`.
- **Testabilidade:** substituir o adapter por um stub de teste não requer modificar `player.gd`.
- **Extensibilidade:** novo dispositivo (gamepad, touchscreen) = novo adapter ou nova implementação de `translate()`.
- **Mapeamento centralizado:** os 5 nomes de ação do `project.godot` ficam em um único lugar (`ALL_ACTIONS`).

## 4. Metodologia
 
A construção desta documentação seguiu três etapas, tomando como fonte primária o código da branch `feat/7-adapter-input`:
 
1. **Levantamento do Adapter.** Leitura de `input_adapter.gd` para mapear os dois modos de operação — event-driven via `translate(InputEvent)` e polling via `poll_move_vector()` e `is_just_pressed()` —, e as 5 constantes de ação que espelham o `project.godot`.
2. **Análise do GameAction.** Inspeção de `game_action.gd` para verificar que `action_name` é `StringName` (sem alocação de heap) e que `strength` cobre tanto teclas digitais (0.0/1.0) quanto analógicos de gamepad (valores intermediários). Confirmação de que `make()` é um factory helper estático que evita boilerplate.
3. **Trace do consumo em player.gd.** Confirmação de que `_read_move_input()` e `_read_dash_input()` — os dois pontos de leitura de input — foram convertidos para chamar o adapter, e que `add_child(_input_adapter)` em `_ready()` registra o adapter na SceneTree sem precisar de autoload.
**Ferramentas utilizadas:** Godot 4 (validação de `Input.get_vector()` vs adapter), Git (branch `feat/7-adapter-input`), Markdown + Docsify (apresentação).
