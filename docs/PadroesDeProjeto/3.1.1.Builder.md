# 3.1.2. Builder (Criacional)

## 1. Introdução

O **Builder** é um padrão criacional do GoF que **separa a construção de um objeto complexo da sua representação final**, permitindo que o mesmo processo de construção produza objetos diferentes. No projeto MadDev, o Builder constrói salas de combate procedurais passo a passo, e o **Director** mantém receitas pré-definidas (sala de combate, descanso, chefão).

### Quando usar

- Quando a construção de um objeto envolve muitos passos opcionais ou ordenados.
- Quando o mesmo "processo" deve produzir variantes diferentes (ex: diferentes tipos de sala).
- Quando o objeto final tem partes que podem ser combinadas de várias formas (inimigos, itens, portas, tilemap).

### RoomBuilder — interface fluente

A classe `RoomBuilder` estende `RoomBuilderBase` e expõe métodos encadeáveis (todos retornam `self`):

```gdscript
class_name RoomBuilder
extends RoomBuilderBase

const _ENEMY_SCENES: Dictionary = {
    &"basic":  "res://scenes/enemies/enemy.tscn",
    &"ranged": "res://scenes/enemies/enemy_ranged.tscn",
}

func set_room_name(name: String) -> RoomBuilderBase:
    _room_name = name
    return self

func add_enemy(enemy_type: StringName, position: Vector2) -> RoomBuilderBase:
    _enemy_list.append({"type": enemy_type, "pos": position})
    return self

func set_exits(directions: Array[Vector2]) -> RoomBuilderBase:
    _door_positions = directions
    return self

func add_item(item_type: StringName, position: Vector2) -> RoomBuilderBase:
    _item_list.append({"type": item_type, "pos": position})
    return self
```

Métodos disponíveis:

| Método | Configura |
|---|---|
| `set_room_name(name)` | Nome da sala (debug/logs) |
| `add_enemy(type, pos)` | Adiciona um inimigo |
| `add_enemies(list)` | Adiciona vários inimigos de uma vez |
| `set_enemy_count(n, type)` | Posiciona N inimigos em posições pré-definidas |
| `add_item(type, pos)` | Adiciona um item consumível |
| `set_exits(directions)` | Define as posições das portas |
| `set_player_start(pos)` | Posição inicial do jogador |
| `set_tilemap(path)` | Recurso do tilemap |
| `build()` | Produz e retorna o `Node2D` final |

### RoomDirector — receitas pré-definidas

```gdscript
class_name RoomDirector
extends RefCounted

var _builder: RoomBuilderBase

func set_builder(builder: RoomBuilderBase) -> void:
    _builder = builder

func build_combat_room(difficulty: int = 1) -> Node2D:
    _builder.set_room_name("Sala de Combate")
    _builder.set_player_start(Vector2(152, 112))
    _builder.set_exits([Vector2(-44, -106)])

    match difficulty:
        1:
            _builder.add_enemy(&"basic", Vector2(100, 100))
            _builder.add_enemy(&"basic", Vector2(200, 100))
        2:
            _builder.add_enemy(&"basic", Vector2(80, 100))
            _builder.add_enemy(&"ranged", Vector2(220, 100))
            _builder.add_enemy(&"basic", Vector2(150, 200))
        3:
            _builder.add_enemy(&"ranged", Vector2(80, 80))
            _builder.add_enemy(&"basic", Vector2(200, 80))
            _builder.add_enemy(&"ranged", Vector2(140, 180))
            _builder.add_enemy(&"basic", Vector2(80, 200))

    if difficulty >= 2:
        _builder.add_item(&"health", Vector2(300, 150))

    return _builder.build()
```

Receitas disponíveis no Director:

| Método | Conteúdo |
|---|---|
| `build_combat_room(difficulty)` | Sala de combate com 2 a 4 inimigos dependendo da dificuldade |
| `build_rest_room()` | Sala de descanso com itens de cura e mana, sem inimigos |
| `build_boss_room()` | Sala do chefão com inimigos `ranged` em flanco e grupo de `basic` |
| `build_empty_room()` | Sala vazia (estrutura mínima, usada em testes) |

## 3. Funcionamento

```gdscript
# Em GameManager (autoload):
var director := RoomDirector.new()
director.set_builder(RoomBuilder.new())

var sala_combate := director.build_combat_room(2)   # difficulty 2
var sala_descanso := director.build_rest_room()
var sala_chefao  := director.build_boss_room()
```

Como o `RoomDirector` mantém só uma referência a `RoomBuilderBase`, é possível **trocar o builder** para gerar a mesma receita com uma representação diferente (ex: um `DebugRoomBuilder` que produz placeholders para testes), sem alterar o Director.

### Benefícios

- **Reutilização:** o mesmo builder constrói qualquer tipo de sala.
- **Flexibilidade:** adicionar uma nova receita é só um novo método no Director.
- **Legibilidade:** a sequência de `add_enemy`/`add_item`/`set_exits` lê quase como prosa.
- **Separação de papéis:** o builder sabe **como** construir partes; o director sabe **quais partes** uma sala precisa.

## 4. Metodologia

A construção desta documentação seguiu três etapas, tomando como fonte primária o código da branch `game`:

1. **Levantamento da hierarquia Builder/Director.** Leitura de `room_builder.gd` (ConcreteBuilder) e `room_director.gd` (Director) para mapear os métodos fluentes e identificar a fronteira de responsabilidades entre os dois papéis.

2. **Análise das receitas.** Inspeção das 4 receitas do Director (`build_combat_room`, `build_rest_room`, `build_boss_room`, `build_empty_room`), incluindo o `match` por `difficulty` em `build_combat_room` que produz salas progressivamente mais difíceis.

3. **Trace da integração.** Confirmação no `game_manager.gd` de que o Director é instanciado com um `RoomBuilder` ao trocar de sala, evidenciando o ciclo Builder → Director → Cliente.

**Ferramentas utilizadas:** Godot 4 (leitura do código), Git (leitura da branch `game`), Markdown + Docsify (apresentação).
