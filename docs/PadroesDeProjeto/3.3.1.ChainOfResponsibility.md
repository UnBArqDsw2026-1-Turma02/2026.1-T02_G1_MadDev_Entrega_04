# 3.3.1. Chain of Responsibility (Comportamental)

## 1. Introdução

O **Chain of Responsibility** é um padrão comportamental do GoF que **passa uma requisição por uma cadeia de handlers**, cada um podendo processá-la, modificá-la ou passá-la adiante. No projeto MadDev, a cadeia é usada para processar dano: `ArmorHandler` → `ResistanceHandler` → `HealthHandler`. O dano inicial passa pela armadura (redução fixa), depois pela resistência (redução percentual) e só então é aplicado à vida do alvo.

### Quando usar

- Quando uma requisição precisa passar por etapas configuráveis e desacopladas.
- Quando se quer evitar `if/else` ou `switch` aninhados que misturam regras heterogêneas (armadura, resistência, status).
- Quando a ordem dos passos deve ser controlada externamente (no Inspector, por exemplo), não no código.

### Estrutura

```
damage_chain (ArmorHandler) ──next──▶ ResistanceHandler ──next──▶ HealthHandler ──next──▶ null

  damage=100        damage=80 (100-20)     damage=40 (80*0.5)     aplica 40 ao HP
  armor=20          resistance=0.5
```

## 2. Implementação no Projeto

### Arquivos

| Arquivo | Papel no padrão |
|---|---|
| `jogo/scripts/damage/damage_handler.gd` | **Handler** abstrato — define `next` e `handle(damage, context)` |
| `jogo/scripts/damage/armor_handler.gd` | **ConcreteHandler** — subtrai `target.defense` (mínimo 0) |
| `jogo/scripts/damage/resistance_handler.gd` | **ConcreteHandler** — multiplica por `(1 - target.resistance)` |
| `jogo/scripts/damage/health_handler.gd` | **ConcreteHandler** — chama `target.apply_final_damage()` |

Os arquivos vivem na branch `game`.

### DamageHandler (Handler abstrato)

```gdscript
@tool
class_name DamageHandler
extends Resource

@export var next: DamageHandler

func handle(damage: int, context: Dictionary) -> int:
    if next != null:
        return next.handle(damage, context)
    return damage
```

A anotação `@tool` permite que o handler seja **montado e visualizado no Inspector** sem precisar rodar o jogo. O campo `next` é `@export`, então pode ser linkado visualmente no editor — exatamente o que o padrão pede: ordem definida fora do código.

### ConcreteHandlers

```gdscript
# ArmorHandler — redução fixa
class_name ArmorHandler
extends DamageHandler

func handle(damage: int, context: Dictionary) -> int:
    var target = context.get("target")
    var armor_value = 0
    if target and "defense" in target:
        armor_value = target.defense
    var reduced_damage = maxi(0, damage - armor_value)
    return super.handle(reduced_damage, context)
```

```gdscript
# ResistanceHandler — redução percentual
class_name ResistanceHandler
extends DamageHandler

func handle(damage: int, context: Dictionary) -> int:
    var target = context.get("target")
    var resistance_pct = 0.0
    if target and "resistance" in target:
        resistance_pct = target.resistance
    var final_damage = damage * (1.0 - resistance_pct)
    return super.handle(final_damage, context)
```

```gdscript
# HealthHandler — terminal: aplica o dano ao HP do alvo
class_name HealthHandler
extends DamageHandler

func handle(damage: int, context: Dictionary) -> int:
    if context.has("target") and context["target"].has_method("apply_final_damage"):
        context["target"].apply_final_damage(damage)
    else:
        print("Dano final calculado, mas nenhum alvo foi especificado: ", damage)
    return super.handle(damage, context)
```

Cada `ConcreteHandler` faz a sua transformação e chama `super.handle(...)` para continuar a cadeia — `super` executa o `handle` do `DamageHandler` base, que delega ao `next`.

### Montagem da cadeia via Inspector

No editor do Godot, a cadeia padrão é montada criando 3 `Resource` em sequência:

1. `armor_chain.tres` (ArmorHandler) com `next = resistance_chain.tres`
2. `resistance_chain.tres` (ResistanceHandler) com `next = health_chain.tres`
3. `health_chain.tres` (HealthHandler) com `next = null`

Cada inimigo (`enemy_base.gd`) e o `Player` recebem a cadeia via `@export var damage_chain: DamageHandler` e a invocam ao tomar dano.

## 3. Funcionamento

```gdscript
# Em enemy_base.gd (resumido):
@export var damage_chain: DamageHandler

func take_damage(amount: int) -> void:
    if not damage_chain:
        apply_final_damage(amount)
        return
    var context = {"target": self}
    damage_chain.handle(amount, context)
```

### Exemplo numérico

Inimigo com `defense = 20`, `resistance = 0.5`, recebe dano de **100**:

| Etapa | Cálculo | Saída |
|---|---|---|
| `ArmorHandler.handle(100)` | `max(0, 100 - 20)` | 80 |
| `ResistanceHandler.handle(80)` | `80 * (1 - 0.5)` | 40 |
| `HealthHandler.handle(40)` | `target.apply_final_damage(40)` | aplica 40 ao HP |

### Benefícios

- **Reordenação trivial:** trocar a ordem (resistência antes da armadura, por exemplo) é arrastar o link no Inspector.
- **Adição não-invasiva:** novo handler (ex.: `ShieldHandler` que absorve N pontos antes de qualquer outra coisa) é só criar um Resource e linká-lo no início da cadeia.
- **Reuso entre Player e Enemy:** a mesma cadeia funciona para qualquer entidade que exponha `defense`, `resistance` e `apply_final_damage()`.
- **Terminação clara:** `HealthHandler` é o último elo e tem efeito colateral (aplica o dano). Os demais são puros — só transformam.

## 4. Metodologia

A construção desta documentação seguiu três etapas, tomando como fonte primária o código da branch `game`:

1. **Identificação da interface.** Leitura de `damage_handler.gd` para mapear a assinatura `handle(damage: int, context: Dictionary) -> int` e o campo `@export var next: DamageHandler` que permite montagem via Inspector.

2. **Análise dos 3 ConcreteHandlers.** Comparação entre `armor_handler.gd` (redução fixa), `resistance_handler.gd` (redução percentual) e `health_handler.gd` (terminal, aplica ao alvo) — confirmando o padrão `super.handle(damage_modificado, context)` que continua a cadeia.

3. **Trace do uso.** Verificação em `enemy_base.gd` de que `take_damage()` cria o `context = {"target": self}` e invoca `damage_chain.handle(amount, context)`, com fallback para `apply_final_damage()` direto caso a cadeia não esteja configurada.

**Ferramentas utilizadas:** Godot 4 (leitura do código e validação no Inspector com `@tool`), Git (leitura da branch `game`), Markdown + Docsify (apresentação).
