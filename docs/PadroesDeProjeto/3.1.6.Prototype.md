# 3.1.4. Prototype (Criacional)
 
## 1. Introdução
 
O **Prototype** é um padrão criacional do GoF que **cria novos objetos copiando uma instância existente (o "protótipo")**, em vez de construí-los do zero. No projeto MadDev, o Prototype resolve o problema de criar cópias de `StudentProfile` em cada run: em vez de reconfigurar os atributos manualmente a cada início de partida, o `StudentProfileRegistry` mantém instâncias-template pré-configuradas e entrega cópias profundas sob demanda via `clone()`. Isso garante que o template original nunca seja acidentalmente modificado durante o jogo.
 
### Quando usar
 
- Quando a criação de um objeto do zero é mais custosa ou complexa do que copiar um existente.
- Quando objetos compartilham uma estrutura base mas precisam de variações independentes por instância.
- Quando o sistema precisa garantir que a instância-template original permaneça imutável.

## 2. Implementação no Projeto
 
### Arquivos
 
| Arquivo | Papel |
|---|---|
| `jogo/scripts/interfaces/i_cloneable.gd` | Interface base — define o contrato `clone() -> Resource`. |
| `jogo/scripts/resources/student_profile.gd` | ConcretePrototype — implementa `clone()` com `duplicate(true)`. |
| `jogo/scripts/resources/student_profile_registry.gd` | Registry/Client — mantém templates e clona sob demanda. Registrado como Autoload. |
 
Os arquivos foram introduzidos na branch `feat/6-prototype-student-profile`.
 
### ICloneable — interface base
 
```gdscript
## Prototype Pattern — interface para objetos que suportam clonagem profunda.
extends Resource
class_name ICloneable
 
## Subclasses DEVEM sobrescrever este método.
func clone() -> Resource:
    return null
```
 
### StudentProfile — implementação de clone()
 
```gdscript
extends ICloneable
class_name StudentProfile
 
@export var profile_name: String = ""
@export var base_hp_modifier: float = 1.0
@export var base_speed_modifier: float = 1.0
@export var base_damage_modifier: float = 1.0
@export var starting_weapon: String = ""
 
func clone() -> Resource:
    assert(profile_name != "", "StudentProfile.clone(): profile_name não pode ser vazio.")
    var copia: StudentProfile = duplicate(true) as StudentProfile
    assert(copia != null, "StudentProfile.clone(): duplicate() retornou null.")
    return copia
```
 
### Atributos do StudentProfile
 
| Atributo | Tipo | Descrição |
|---|---|---|
| `profile_name` | `String` | Identificador único do perfil (ex: `"guerreiro"`) |
| `base_hp_modifier` | `float` | Multiplicador de HP máximo do jogador |
| `base_speed_modifier` | `float` | Multiplicador de velocidade de movimento |
| `base_damage_modifier` | `float` | Multiplicador de dano base |
| `starting_weapon` | `String` | Caminho do Resource da arma inicial |
 
### StudentProfileRegistry — templates e clonagem
 
```gdscript
extends Node
## class_name omitido: script é registrado como Autoload "StudentProfileRegistry".
## Autoload e class_name com o mesmo nome causam erro no Godot 4.
 
var _templates: Dictionary = {}
 
func _ready() -> void:
    _register_default_templates()
 
func clone_profile(profile_name: String) -> StudentProfile:
    assert(_templates.has(profile_name), "Perfil '%s' não encontrado." % profile_name)
    return _templates[profile_name].clone() as StudentProfile
```
 
### Perfis pré-configurados
 
| Perfil | HP | Velocidade | Dano |
|---|---|---|---|
| `"guerreiro"` | +30% | -10% | +20% |
| `"explorador"` | base | +30% | -10% |
| `"mago"` | -20% | base | +50% |

## 3. Funcionamento
 
```gdscript
# Em GameManager — ao iniciar uma run com o perfil escolhido pelo jogador:
var perfil := StudentProfileRegistry.clone_profile("guerreiro") as StudentProfile
perfil.apply_to(player)
 
# Modificar o clone NÃO afeta o template original:
perfil.profile_name = "guerreiro_run_42"
perfil.base_hp_modifier = 1.5   # apenas esta instância muda
```
 
Para adicionar um novo perfil, basta criar e registrar um `StudentProfile`:
 
1. Instanciar um `StudentProfile` com os atributos desejados.
2. Chamar `StudentProfileRegistry.register_template(perfil)`.
3. O novo perfil fica disponível via `clone_profile("nome_do_novo")`.
### Benefícios
 
- **Imutabilidade do template:** `duplicate(true)` garante cópia profunda — o original nunca é alterado.
- **Extensibilidade:** adicionar um perfil não requer modificar nenhum código consumidor.
- **Validação antecipada:** `assert` em `clone()` detecta templates inválidos antes de iniciar a run.
- **Compatibilidade com Autoload:** sem `class_name` no registry, evita conflito nativo do Godot 4.

## 4. Metodologia
 
A construção desta documentação seguiu três etapas, tomando como fonte primária o código da branch `feat/6-prototype-student-profile`:
 
1. **Levantamento da hierarquia Prototype.** Leitura de `i_cloneable.gd` (interface base), `student_profile.gd` (ConcretePrototype) e `student_profile_registry.gd` (Registry/Client) para mapear os contratos, a cadeia de herança `ICloneable → StudentProfile` e o mecanismo de `duplicate(true)` para cópia profunda de Resources do Godot 4.
2. **Análise da integração com Autoload.** Identificação do conflito `class_name` vs Autoload no Godot 4, documentado diretamente no código como comentário explicativo, e verificação de que `StudentProfileRegistry.clone_profile()` é o único ponto de acesso aos perfis — nenhum código externo instancia `StudentProfile` diretamente.
3. **Trace da aplicação ao jogador.** Confirmação de que `apply_to(player)` usa `player.get("atributo") != null` para aplicar os modificadores de forma segura, sem acoplar o perfil à implementação específica do `player.gd`.
**Ferramentas utilizadas:** Godot 4 (validação do comportamento de `duplicate(true)` e teste de Autoload), Git (branch `feat/6-prototype-student-profile`), Markdown + Docsify (apresentação).
